# OTA updates, versions & the fleet

Over-the-air updates let you ship firmware to deployed pucks without USB. This page is the
**device/operator** view; the **server** side (where the files live) is in
[backend.md](backend.md) and the full runbook is [`tools/OTA-SETUP.md`](../tools/OTA-SETUP.md).

## How a puck updates

A core-0 background task ([`Ota` in `src/services.cpp`](../src/services.cpp)) polls a JSON manifest
over HTTPS (CA-pinned to `MQTT_CA_CERT`, never `setInsecure`). Three triggers funnel into one check:

1. **Boot + every `OTA_POLL_HOURS`** (default 24h).
2. **MQTT broadcast** `puck/all/ota` — an instant "re-check now" nudge.
3. **Settings → Check Updates** button.

When the manifest's integer `version` is **greater than** the compiled `FW_VERSION`
([`src/version.h`](../src/version.h)), the puck pops a full-screen **Update / Later** overlay; on
confirm it flashes the inactive OTA slot (dual 6.5 MB app slots, `default_16MB.csv`) and reboots. A
failed download leaves the running firmware untouched. **No auto-rollback** for a bad-but-bootable
image — bench-test before publishing. Blank `OTA_MANIFEST_URL` = OTA off.

> The on-screen status maps fetch results: reached-but-nothing-newer → "up to date"; couldn't connect
> → "server unreachable". A manifest `version` of 0 is treated as a baseline and never offered.

## Per-board isolation — each puck only sees its own firmware

A CoreS3 image would brick a Waveshare (wrong drivers → black screen). So OTA is **board-scoped by
construction**: the device builds every URL from its compiled `PUCK_OTA_SUFFIX`
(`boards/<id>/board_config.h`), so it can't even *name* another board's files —
`version{suffix}.json`, `versions{suffix}.json`, `firmware{suffix}-v<N>.bin`. The incumbent CoreS3
uses `""` (the existing unsuffixed paths → no fleet disruption); other boards get `-<board>`. As
defence in depth, `fetchManifest` also refuses a manifest whose optional `"board"` field is present
and ≠ `PUCK_BOARD_ID`.

On the **CI** side this is matched by a build **matrix**: `release.yml` builds every board and
publishes each board's suffixed `firmware{suffix}-v<N>.bin` / `version{suffix}.json` (carrying its
`"board"`) / `versions{suffix}.json`. One global `FW_VERSION` / `fw-v<N>` tag covers all boards. (The
Waveshare bin builds + publishes as a **stub** until its drivers exist.)

## Targeted & selectable installs

Beyond the latest-for-all manifest:

- **On-device picker** — Settings → Versions lists installable versions (from `versions{suffix}.json`,
  regenerated by CI from the bins on disk) and can install or **downgrade** to any of them. The device
  builds the bin URL from the compiled base, so only a *published* version can ever be picked.
- **Targeted push** — an operator publishes `fleet/ota/<code>` `{"v":N,"force":0|1}` (`force` flashes
  silently, else it pops the confirm). Sent via `tools/fleet.py send <code|all|kind> <ver> [--force]
  [--board B]` — `all`/a kind fans out (optionally restricted to one board), and each device resolves
  its own board's bin, so a push is board-safe.

## Version channels (who sees release candidates)

The picker filters by channel so prod devices don't see in-progress RCs:

- **prod** (default) → only **released** versions (those with a final `fw-v<N>` tag), `≥ min`.
- **test/beta** → the full list incl. RCs; the picker header reads `Firmware (beta)`.

A device's **kind** is its retained `fleet/channel/<code>` marker — an arbitrary label (`test`/`prod`/
`xyz`/…). Visibility is binary: `test`/`beta` see all builds, every other kind sees released only
(picker header reads `Firmware (beta)` for test). The operator sets it with
`tools/fleet.py channel <code> <kind>` — **no public device list, no codes in logs**. `OTA_MIN_VERSION`
(CI variable) is the floor.

## Fleet management & the repo split

Fleet ops print/target **device codes** (= MQTT usernames), so they're split by repo:

- **Public `planepuck-fw`** *builds* (the per-board matrix) + does the leak-safe build-time nudges
  (final broadcast, RC→test push, `--quiet` so no codes hit the public log).
- **Private `planepuck-ops`** runs the code-revealing management — `tools/fleet.py list
  [--board B] [--kind K] [--json]`, `send`, `channel`, `broadcast` — in **private** Actions logs.
  Template + setup: [`tools/ops/README.md`](../tools/ops/README.md). `fleet.py` itself stays public
  (no secrets); its core is `--json`/importable so a future web frontend can reuse it.
