# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

PlanePuck is ESP32-S3 firmware for the **M5Stack CoreS3** — a touchscreen "app gadget" platform. It boots into a launcher that hosts small apps (Clock, Emoji Ping, Weather, Flight, Friends, Settings) backed by always-on services (time, auto-dimming, notifications, weather/flight fetchers, compass, friends/MQTT messaging). It runs **fully offline** by default; Wi-Fi/timezone/ZIP are set on-device at first boot (captive portal), while MQTT and other options live in `src/config.h`.

Built with PlatformIO + Arduino framework. Display/touch/audio/RTC are all accessed through the `M5Unified` library (`M5.Display`, `M5.Touch`, `M5.Speaker`, `M5.Rtc`).

## Build & flash

This is a PlatformIO project — there is no test suite or linter. Build/upload via the PlatformIO VS Code buttons, or the CLI:

```bash
pio run                    # build (env: m5stack-cores3)
pio run -t upload          # flash to a connected CoreS3
pio device monitor         # serial monitor @ 115200 baud
pio run -t clean           # clean build artifacts (.pio/)
```

First build is slow — it downloads `M5Unified`, `PubSubClient`, and `ArduinoJson` (declared in `platformio.ini`).

## Architecture

Three layers, each in its own file under `src/` (the bare filenames below all live there). The dependency direction is strict: **apps depend on services; services and the main loop never depend on apps** (except `main.cpp`, which registers them).

- **`main.cpp`** — owns `setup()`/`loop()`, the launcher UI (a radial ring of circular chips titled with the user's `Settings::displayName` or "PlanePuck" if unset; `launcherChipPos()` computes positions for both draw and hit-test, and `drawAppIcon()` paints a small anti-aliased vector glyph per app, keyed by app name), and the back chip. The loop runs at ~60 fps: it samples exactly one tap per frame into the global `gTap`, pumps every service's `loop()`, then dispatches to the active app (or the launcher when `active == nullptr`). Incoming MQTT goes through `mqttRouter` → `Friends::handleSocial()` first (the always-on friends protocol), then falls through to `active->onMqtt()` for anything it doesn't consume; a confirmed-friend ping pops a full-screen emoji overlay (`drawPing`/`gPing*`) over whatever screen is showing, with **Reply** (opens the Emoji app aimed at the sender) and **Mute 1h** buttons.

- **`App.h`** — the `App` abstract base (`onEnter`/`loop`/`onExit`/`onMqtt`, plus `needsNet()` — apps that require Wi-Fi return true so `main.cpp` shows a tappable "Finish setup" prompt instead of an empty screen until Wi-Fi is configured) and the `Tap` struct. **To add an app:** subclass `App`, then instantiate it and add it to the `apps[]` array in `main.cpp` — that array drives both the launcher list and indexed dispatch. Nothing else needs to change.

- **`apps.h`** — concrete apps (`ClockApp`, `EmojiApp`, `WeatherApp`, `FlightApp`, `FriendsApp`, `SetupApp`), header-only. Apps own their own drawing and read services; they redraw lazily (Clock on second-change or drift-step, Emoji only when `dirty`) to avoid flicker. The screens that repaint on a **periodic data refresh** — Clock (band), Weather (single + grid), and Flight (list / detail / radar) — each **compose off-screen into a PSRAM `M5Canvas` and push in one blit** (via a per-app `g` draw-target pointer / `scope` sprite), so a ~15–60 s refresh never flashes; the interactive screens (keyboards, Emoji picker, Friends home, Settings) repaint only on a tap/event, so they draw directly. `ClockApp` shows an optional user-set **name** at the top (`Settings::displayName`, chosen in the captive portal; blank = hidden), the big local time that slowly **drifts** position (anti-burn-in) plus up to `MAX_WORLD_CITIES` **world-clock** rows (cities chosen in the captive portal; rendered via `WorldClock::timeFor`, which briefly swaps the global TZ under a mutex shared with `ClockService::getTime` since the Weather task also reads local time); `main.cpp` auto-opens it from the launcher after `LAUNCHER_IDLE_MS` of idle. The time+status band is composed off-screen in an **`M5Canvas` (PSRAM) and pushed in one blit** — flicker-free even on a drift step; the fixed city rows are drawn directly below it. `WeatherApp` is display-only — it reads the `Weather` service's background-updated cache and never fetches on the UI thread; with one location (home, or a single configured city) it shows the big detailed view, and with 2–`MAX_WEATHER_CITIES` configured cities it switches to an equal-row grid (`drawGrid`: city + temp + condition per row). `FlightApp` is a small mode machine (LIST / RADAR / DETAIL / SEARCH): radar plots `dist`/`bearing` as dots around center, with nearby airports as hollow-square reference markers (two-finger **pinch-to-zoom** — out to `RADAR_MAX_NM` (150nm), and zooming out **widens the live fetch** so farther planes/airports load (`Flight::setRange()`, debounced to gesture-end; reset on app exit) — + one-finger **drag-to-re-center** (the range rings **and** crosshair stay *fixed* at the screen center `w/2,116`; only the world — your-location dot, planes, airports, plotted at the panned origin `w/2+panX, 116+panY` — slides under them, so a drag moves the scope's center off *you* onto a new spot and the rings then read distance from there, letting you place the crosswire over nearby cities/airports); a RESET button clears zoom+pan; rotates to **heading-up** when the `Compass` service has a calibrated magnetometer, otherwise north-up — the radar reads `M5.Touch` directly since `gTap` carries only one point, and a drag is distinguished from a tap via `wasClicked()`; the whole scope is **composed off-screen in a 320×240 `M5Canvas` (PSRAM, the `scope` member, draws via the `g` target pointer) and pushed in one blit**, so a ~15s data refresh or a heading turn never flickers — the figure-8 cal screen is the one part drawn directly), search is an on-screen keyboard whose `GO` auto-detects the input: an all-letters **3/4-char airport code** (IATA/ICAO) calls `Flight::searchAirport()` (async Overpass lookup → `gCenterOverride` makes the nearby fetch query the **airport's** coords) and opens an **airport-centered** radar (airport-square at center, planes around it, a `you` reference marker, code/name overlay); anything else is a **flight number** run through `Flight::toCallsign()` (IATA→ICAO, e.g. SQ31→SIA31) then `Flight::track()`, opening the radar **centered on that plane** (forced north-up): the searched aircraft sits at the scope center with a heading tick, airports near *it* are the reference markers (a separate Overpass fetch — `Flight::planeAirports()`), a `you` marker shows your bearing/distance from it (reciprocal of `tp.bearing`), and a details overlay (callsign / route / altitude / speed / climb-descend trend / distance, via `drawTrackedOverlay`) sits on a solid card **below the back chip** (top-left, larger font so it clears the chip and reads over the rings); the back (`<`) chip steps RADAR→LIST→launcher (`App::onBack`), and tapping a *nearby* plane opens its detail (generous hit target; a miss does nothing — no accidental exit). So **RADAR has three centers**: you-centered over the nearby snapshot normally, plane-centered when a searched flight is tracked (`searched && Flight::tracked()`), and airport-centered when an airport code was searched (`aptMode && Flight::airportCenter()`). Detail (tap a *nearby* plane) shows route + a climb/descend arrow from `vrate`. `EmojiApp` sends an emote — drawn as a vector glyph via the shared `drawEmote()`, not text — to the remembered recipient (a specific friend or all) through `Friends`. `FriendsApp` manages friends: shows your code to share, adds a friend by code (hex keypad), approves/denies incoming requests, lists/removes friends.

- **`services.h` / `services.cpp`** — fourteen always-on subsystems, each a namespace, all initialized in `setup()` (Weather/Flight/Ota spin up background tasks; the rest are pumped from the per-frame `loop()`):
  - `Settings` — persisted user config (Wi-Fi creds, timezone, weather ZIP) in NVS via `Preferences` (namespace `"puck"`). On-device values override the `config.h` compile-time defaults; `Net`, `ClockService`, and `Weather` read through it. Must be initialized first in `setup()`.
  - `Provision` — on-device settings: a SoftAP (`PlanePuck-Setup`) + captive portal (built-in `WebServer` + `DNSServer`, no extra lib_deps) at `192.168.4.1`. The form takes a display **name** (shown on the Clock + to friends), scans networks, takes a password, a timezone dropdown, weather cities, and a ZIP code; on save it writes to `Settings` and calls `ESP.restart()`. Surfaced via `SetupApp` ("Settings" in the launcher), which `main.cpp` auto-enters on first boot when `Settings::haveWifi()` is false.
  - `Net` — Wi-Fi; reads creds from `Settings`, no-op when none configured (offline mode).
  - `ClockService` — seeds time from the RTC on boot, then upgrades to NTP once `Net::online()`. `synced()` reflects whether NTP has taken over.
  - `Dim` — auto-brightness with a **runtime-detected** light sensor. At boot it probes the LTR-553 (internal I2C `0x23`, reads `MANUFAC_ID 0x87` == `0x05`); if present, brightness follows ambient (raw CH0 mapped `LUX_DARK`..`LUX_BRIGHT` → `BRIGHT_NIGHT`..`BRIGHT_DAY`), otherwise it falls back to the time-of-day night window — so one binary serves both the full CoreS3 and the sensor-less SE (`FORCE_NO_LIGHT_SENSOR` forces time-based). Three tiers by idle time: `< WAKE_FULL_MS` (60s) after a touch = full (`wake()`), then the normal ambient/time level, then after `IDLE_DIM_SEC` a **deep idle-dim** (ambient × `IDLE_DIM_SCALE_PCT`, or a fixed `BRIGHT_IDLE_DAY`/`BRIGHT_IDLE_NIGHT` that's lower at night). `cur` still creeps toward the target 1 step/frame. **Auto-dim is Clock-only**: `main.cpp` calls `Dim::allowIdleDim(!active || active->dimsWhenIdle())` each frame and only `ClockApp` overrides `dimsWhenIdle()`→true, so apps you're actively using (Weather/Flight/etc.) hold at the normal ambient/time level and never idle-dim — only the Clock (the idle/screensaver face) and the launcher dim. While the Clock is in the deep idle-dim, `dimmed()` is true and `main.cpp` swallows the first touch (it only wakes the screen). `BRIGHT_NIGHT`/`BRIGHT_IDLE_NIGHT` are raised (64/32) so sensor-less units aren't too dim at night (this also raises the sensor's dark-room floor).
  - `Compass` — tilt-compensated heading from the IMU magnetometer; drives `FlightApp`'s heading-up radar. **Auto-detects** the magnetometer via `M5.Imu.getImuInstancePtr(1)` (null on units without one → `available()` false → radar stays north-up; the standard CoreS3 IMU is the 6-axis BMI270, so a mag isn't guaranteed). Heading uses a horizontal-plane vector projection (robust to the accel sign convention; the device's top edge is "forward"). Figure-8 calibration runs through M5Unified's built-in offset system (`setCalibration`/`saveOffsetToNVS`/`loadOffsetFromNVS`), with a `magcal` flag in `Settings` marking it done. `HEADING_OFFSET_DEG` (axis fix) and `MAG_DECLINATION_DEG` (magnetic→true) in `config.h` tune the output. **No GPS on the CoreS3** — the radar center is the `Geo` (IP/ZIP) location, not a live fix.
  - `Geo` — shared location resolver (saved ZIP via Zippopotam.us → IP via ip-api.com → `WEATHER_*` config.h fallback), cached after first success. Used by both `Weather` and `Flight`; call only from a background task (it blocks on HTTP).
  - `Flight` — aircraft data via a **FreeRTOS background task** (core 0, larger stack for TLS). Each ~15s cycle does up to three things: (1) **nearby** list from adsb.lol `/v2/lat/lon/dist` over HTTP (search radius = `gFetchNm`, default `FLIGHT_RADIUS_NM` but widened by the radar's zoom-out via `setRange()`; on-ground / `alt <= 0` aircraft dropped, then sorted by distance, nearest `FLIGHT_KEEP` kept); (2) if a callsign is being **tracked** (`track()`), its live position from `/v2/callsign/{cs}` (distance/bearing computed locally via `relTo()` since that feed lacks `dst`/`dir`); (3) a pending **route** lookup from **OpenSky Network** (free, OAuth2), keyed by the aircraft's **ICAO24 hex** (captured into `Plane.hex` from the adsb.lol feed; resolved from the requested callsign by `hexForCallsign()`). An OAuth2 *client-credentials* bearer token is fetched from `auth.opensky-network.org` and cached (`osEnsureToken()`, refreshed ~1 min before expiry), then `/api/flights/aircraft?icao24=&begin=&end=` is read and the most recent flight that actually has airport data wins, mapping `estDepartureAirport`/`estArrivalAirport` (ICAO codes; a US `K###` is shown as the 3-letter IATA-ish code via `dispAirport()`). Needs `OPENSKY_CLIENT_ID`/`OPENSKY_CLIENT_SECRET` in `config.h` (blank → "route unknown"); TLS bounded by `setHandshakeTimeout(8)`; cached per-callsign (incl. negative cache). **Window gotcha:** OpenSky rejects a `begin..end` span that touches more than **2 day-partitions** (HTTP 400 "query across 2 partitions" — which silently made *every* route "unknown"). `time()` is UTC epoch, so `begin` is pinned to 00:00 UTC of the previous day (`now - now%86400 - 86400`) → the window always touches exactly 2 UTC days (24–48h of history). **Route semantics:** OpenSky returns the *in-progress* flight too, with its **departure** known but **arrival null** until the plane lands — so an overhead plane shows `ORIG > ?` (where it came from), while a just-landed or searched flight shows the full `ORIG > DEST`. There's no live *destination* for en-route traffic anywhere in free data (that's flight-plan data). (The earlier free static route DBs adsbdb/hexdb were dropped as unreliable: callsign→route maps go stale and numbers get reused, which produced wrong origins/destinations.) All parses use ArduinoJson filters to bound memory. It also fetches **airports** (IATA code + position) from the OpenStreetMap **Overpass API** (free, no key; plain HTTP, raw OQL `nwr[aeroway=aerodrome][iata](around:...)` POST body) via the shared `fetchAirportsAt(lat,lon,...)` — **once near you** (`airports()`, the reference markers on the nearby radar), and, while a flight is **tracked**, a second set **around the tracked plane** (`planeAirports()`, the markers on the plane-centered radar): re-fetched when the plane moves >25nm, with each marker's dist/bearing recomputed relative to the live plane each cycle via `relFrom()` (the generalized `relTo()` that takes an arbitrary origin). UI reads via `snapshot()` / `airports()` / `planeAirports()` / `tracked()` / `route()` / `version()`. **No ETA/time-to-destination** (free ADS-B + route data has no schedule).
  - `Weather` — live conditions from Open-Meteo over plain HTTP (parsed with ArduinoJson; HTTP avoids TLS handshake stalls that were hanging the fetch). A **FreeRTOS background task** (pinned to core 0, started by `begin()`) refreshes every 60s — or immediately on `refreshNow()` — so the blocking HTTP never touches the UI thread. **Multi-city:** if weather cities are configured in the captive portal (up to `MAX_WEATHER_CITIES`, each **free-typed as a city name or US ZIP** — not the timezone list), the task **geocodes each on-device once** (`geocodeQuery`: a 5-digit ZIP → Zippopotam, otherwise a name search via Open-Meteo's geocoder, both plain HTTP + ArduinoJson-filtered; result cached on the `WCity`) — for same-named cities it picks the **most-populous** hit, and a trailing `City, CC` qualifier (e.g. `Kochi, IN`) forces a country — then fetches by coordinate (`fetchInto`) and the UI reads them via `count()` / `get(i,out)`; with none configured it fetches the **home** location (saved **ZIP** → **IP** → `WEATHER_*` fallback) into reading 0. The UI also has `latest(out)` (home/first) / `updating()` / `version()`; the single-city view shows the reading plus the wall-clock fetch time (`Reading::atHour`/`atMin`, `updated H:MM`) and repaints only when `version()` changes.
  - `Notify` — `gentle()` = soft beep (muted during the night window) + dot; `alert()` = always-audible beep used for friend pings (ignores night). The dot overlay is drawn on every screen, including the launcher. `mute(ms)`/`unmute()` silence beeps for a window (sound only — the ping overlay still shows); the ping's right button toggles **Mute 1h** ↔ **Unmute**. (The CoreS3 has **no vibration motor** — sound + dot are the only alerts.)
  - `Broker` — MQTT via `PubSubClient`, optionally TLS (`MQTT_TLS`: `WiFiClientSecure` CA-pinned to `MQTT_CA_CERT`). No-op when `MQTT_HOST` is blank. **The blocking TLS connect runs on its own FreeRTOS task, never the UI loop** (a blocking connect there froze touch); a `gUp` flag hands the `client` between the task (connects only while disconnected) and the UI thread (`loop()`/`publish()` only while connected), lock-free. Client id is unique per chip; the **MQTT username is the device's friend code** and the **password is per-device in NVS**, never compiled in — either typed in the captive portal or, when `ENROLL_URL`/`ENROLL_TOKEN` are set, **self-enrolled**: a fresh puck (no saved password) generates a random one and `POST`s `{code,pw}` over HTTPS (bearer `ENROLL_TOKEN`, CA-pinned to `MQTT_CA_CERT`) to the droplet's enroll endpoint, which runs `mosquitto_passwd`; the puck saves the password to NVS and connects — zero-touch, all inside `Broker`'s connect task (`enrollConfigured()`/`enrollOnce()`), gated on `ClockService::synced()` (TLS needs a real clock). Server side: `tools/enroll-server.py` (+ `planepuck-enroll.service`, `enroll.Caddyfile`).
  - `Friends` — social layer for emoji: add friends by code with **mutual approval**; only mutually-confirmed friends exchange emotes (others dropped). Always-on, owns inbound routing via `handleSocial()` so requests/pings work on any screen. Identity = an 8-hex **friend code** from `ESP.getEfuseMac()` (= the MQTT username) + an editable display name (your own, sent to friends in `req`/`ok` payloads). Each friend may also be given a **local nickname** (`Friend.nick`, NVS key `k` in the list blob, **never published**); the UI shows `nick → their name → code` everywhere (`labelFor()`), settable from the Friends page (tap a friend's row) or when **adding** (after the code). Incoming requests render two lines — the sender's (spoofable) **name** + their **device id** — so you verify identity before a **one-tap OK** accept. List persisted as JSON in NVS. One wildcard inbox `puck/<code>/#`; messages `puck/<dst>/<type>/<src>` (type ∈ `req`/`ok`/`no`/`emo`), requests/approvals **retained** (offline delivery, cleared after handling), emotes not. SENT/RECEIVED/CONFIRMED state machine, auto-confirming when both add each other.
  - `Ota` — over-the-air firmware updates. A **core-0 background task** pulls a JSON manifest (`OTA_MANIFEST_URL`) over HTTPS (`WiFiClientSecure` pinned to `MQTT_CA_CERT` — the droplet's Let's Encrypt cert; gated on `Net::online() && ClockService::synced()`), compares the manifest `version` against the compiled `FW_VERSION`, and when newer raises an `AVAILABLE` phase. `main.cpp` pops a full-screen **Update / Later** overlay (modeled on the ping overlay); on confirm, `doFlash()` suspends the Weather/Flight/Broker tasks, runs arduino-esp32 `httpUpdate` (writes the inactive OTA slot, `onProgress`→`percent()` for the on-screen bar), and `ESP.restart()`s into the new image. Three triggers funnel into one `checkNow()`: boot + every `OTA_POLL_HOURS` (scheduled in `loop()`), an `OTA_PUSH_TOPIC` (`puck/all/ota`) MQTT broadcast (routed in `mqttRouter` independent of the active app — needs the multi-subscription `Broker` + a Mosquitto `topic read` ACL), and the Settings "Check Updates" button. Tapping **Later** sets a **RAM-only** 24h snooze (`Ota::gSnoozeUntil`, *not* persisted — so any reboot, including the post-flash `ESP.restart()`, clears it); the automatic checks (boot + poll + push) honor it so it won't re-nag within the session, while the Settings button passes `checkNow(true)` to bypass it (manual update any time). A failed download leaves the running firmware untouched. Off when `OTA_MANIFEST_URL` is blank. **Pucks already ship OTA-capable** (`default_16MB.csv` = dual 6.5 MB app slots, pinned in `platformio.ini`); the partition table can't be changed over OTA, so never regress that. **No auto-rollback for a bad-but-bootable image — bench-test before publishing.** Operator side: `tools/release.sh` + `tools/OTA-SETUP.md`. **Beyond the latest-for-all manifest, OTA also does *targeted/selectable* installs**: `pushVersion(v, force)` builds the URL from a compiled `OTA_BIN_BASE` + version (so only a *published* binary can be picked, never an arbitrary URL) and flashes **any** version incl. **downgrades** — `force` flashes silently, else it pops the same confirm overlay. It's reached three ways: a per-device MQTT command `fleet/ota/<code>` (`{"v":N,"force":0|1}`, routed in `mqttRouter` → `onFleetCmd`), and an on-device **Settings → Versions** picker (`requestVersions()` fetches `OTA_VERSIONS_URL` = `versions.json`, which CI regenerates from the bins on disk). **Channel filtering:** prod devices see only `released` versions, test/beta devices see the full list incl. RCs (both floored at `min`); a device is "beta" when it receives a **retained, authenticated** `fleet/channel/<code>`=`"test"` marker (`Ota::onChannel`, cached in NVS via `Settings::beta`) — set by the operator (`fleet.py sync-channels` / the `sync-channels` Action), never a public device list (this replaced the old public `OTA_TEST_URL`/`test-devices.json`, which is now unused).
  - `Fleet` — fleet telemetry so an operator can *see* devices. On each MQTT (re)connect it publishes **retained** `fleet/status/<code>` (`{"v":FW_VERSION,"n":name}`) + `fleet/online/<code>`="1"; a `Broker::setWill()` LWT flips `fleet/online/<code>`→"0" on a drop (online/offline). UI-thread publishes only (in `Fleet::loop()`, gated on a `Broker::connected()` rising edge). Operator reads it + sends targeted pushes via **`tools/fleet.py`** (`list`/`send <code> <ver> [--force]`/`broadcast`, a `mosquitto_*` wrapper). New Mosquitto ACLs (in `OTA-SETUP.md`): devices `pattern write fleet/status|online/%u` + `pattern read fleet/ota|channel/%u`; operator `topic read fleet/#` + `topic write fleet/ota|channel/+`.

## Configuration

**Wi-Fi and timezone are configured on-device at first boot** (see `Provision`/`Settings`) and persisted to NVS — not by editing `config.h`. `config.h` holds compile-time `#define`s for everything else (MQTT broker host/port + `MQTT_TLS`/`MQTT_CA_CERT`, `DEVICE_ID`, brightness levels, night window, light-sensor flag, flight radius, compass corrections `HEADING_OFFSET_DEG`/`MAG_DECLINATION_DEG`, OTA `OTA_MANIFEST_URL`/`OTA_POLL_HOURS`/`OTA_PUSH_TOPIC`, zero-touch enroll `ENROLL_URL`/`ENROLL_TOKEN`, and screensaver/world-clock/weather knobs `LAUNCHER_IDLE_MS`/`CLOCK_DRIFT_*`/`IDLE_DIM_*`/`MAX_WORLD_CITIES`/`MAX_WEATHER_CITIES`) plus *fallback* defaults for Wi-Fi/timezone. World-clock + weather cities are picked on-device in the captive portal (Settings). **`config.h` is gitignored** (it holds the OpenSky secret); a fresh clone copies `src/config.h.example` → `src/config.h`, and the integer **`FW_VERSION` lives in the tracked `src/version.h`** (so the release commit/tag can version it). The per-device MQTT password lives in NVS, **not** in `config.h` — entered via the captive portal, or **self-provisioned** by the optional zero-touch **enroll** endpoint (`ENROLL_URL`/`ENROLL_TOKEN`: a fresh puck registers a random password with the droplet, so recipients never type one — see `Broker` above and `tools/OTA-SETUP.md`). **First burn:** OTA can't flash blank silicon, so a fresh CoreS3 is flashed once over USB — easiest via the **ESP Web Tools** browser installer at `https://fw.example.com/` (CI publishes `tools/webinstall/index.html` + `manifest.json` + `planepuck-merged.bin` next to the OTA files); after that, OTA takes over. Empty string values are the "feature off" sentinel: no Wi-Fi configured → offline, blank `MQTT_HOST` → local-only, blank `OTA_MANIFEST_URL` → no OTA. **Shipping updates (CI):** `tools/release.sh ["notes"]` only bumps `src/version.h`, commits, and **pushes a `fw-v<N>` tag** (auto-bump = latest tag+1, or pass a version). A tag-triggered **GitHub Action** (`.github/workflows/release.yml`) then rebuilds — reconstructing `config.h` from `config.h.example` + **secrets in the Actions vault** (`OPENSKY_CLIENT_ID/SECRET`, `DEPLOY_SSH_KEY`, `DROPLET_HOST`, optional MQTT operator creds + optional `ENROLL_TOKEN`) — and publishes the OTA bin + manifest **plus the merged image + ESP Web Tools installer page** to the droplet (so `config.h`'s secrets never leave the vault). Setup + secret list: `tools/OTA-SETUP.md` (CI). Pucks auto-poll / accept an MQTT push and apply after an on-screen confirm; tapping **Later** snoozes the prompt 24h (manual check via Settings always works). *(The published `.bin` still embeds the OpenSky secret — downloadable from the unlisted fw URL; low-value, proxy via the droplet if it matters.)*

Emoji messaging is **friends-based**, not broadcast — `GROUP_ID` is deprecated/unused. For a private broker, point `MQTT_HOST` at its **domain** (must match the TLS cert) with `MQTT_TLS 1` / port 8883; the Let's Encrypt root(s) are embedded as `MQTT_CA_CERT`. Each device's MQTT username is its auto friend code and its password is entered on-device. The intended server is a self-hosted Mosquitto with per-device `mosquitto_passwd` entries and a `%u` pattern ACL (a device reads only `puck/%u/#` and may publish only as itself). For a non-TLS test, set `MQTT_TLS 0` / port 1883.

## Conventions

- Emote **codes** are ASCII (`"<3"`, `":)"`) — the MQTT wire format and default-font fallback — but the picker and the incoming-ping overlay draw them as **anti-aliased vector graphics** (`drawEmote()` in `apps.h`); launcher app icons too (`drawAppIcon()` in `main.cpp`).
- Apps consume a tap by setting `gTap.pressed = false` after handling it. The Flight radar is the one exception: it reads `M5.Touch` directly (two-finger count + `getDetail(i)`) because `gTap` carries only a single point, which can't express a pinch.
- Drawing is immediate-mode against `M5.Display`; keep redraws gated (dirty flags / change detection) to avoid flicker on the 60 fps loop.
- **Never block the ~60 fps loop.** Network/HTTP/TLS work belongs on a FreeRTOS background task (`Weather`, `Flight`, `Broker`); a blocking call on the UI loop freezes touch — this is exactly what the MQTT TLS connect did before it was moved to its own task.
- **`Serial.printf` goes to UART0 (physical pins), not USB.** Over the USB port (`pio device monitor` / USB-serial-JTAG) you only see ESP-IDF `[E]`/`log_e()` output, not the `[weather]`/`[flight]`/`[mqtt]` `Serial` logs — use `log_e()` for diagnostics you need to read over USB.
- **The captive portal keeps a blank Wi-Fi password** (`Provision::handleSave` only rewrites Wi-Fi when a password is typed; blank = keep saved). Same "blank = keep" for the MQTT password, so editing one field never wipes another's secret.
